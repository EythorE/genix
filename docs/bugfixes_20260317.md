# Bug Fixes — 2026-03-17

## Bug 1: Multi-stage pipes produce garbage output (CRITICAL)

### Symptoms

```
# echo hi | cat | grep hi
"o�(o���Jg�./
```

Three-stage (or more) pipelines produced binary garbage instead of the
expected output. Two-stage pipelines (`echo hi | cat`) worked fine.
After running broken pipelines, subsequent commands would sometimes fail
with "Out of space" errors due to heap corruption.

### Root Cause

**Design flaw in dash's vfork integration with forkshell().**

On Genix, `fork()` is `#define`d to `vfork()` (in `genix_compat.h`)
because there is no MMU for copy-on-write. The vfork child shares the
parent's entire address space until it calls `exec()` or `_exit()`.

`forkshell()` in `apps/dash/jobs.c` called `fork()` (→ `vfork()`)
**without** setting the `vforked` flag. This meant `forkchild()` ran
its full initialization path in the parent's address space:

1. **`forkreset()`** — cleared `redirlist`, `handler`, trap strings
2. **`freejob()` on all current jobs** — freed the pipeline's `procstat`
   array back to the heap

When `vfork_restore` resumed the parent:
- `forkparent()` wrote to `jp->ps[jp->nprocs++]` — **use-after-free**
  (procstat was freed by the child)
- `waitforjob()` later called `freejob()` again — **double-free**,
  corrupting the heap allocator's metadata
- Subsequent children's `malloc()`/`free()` cycles reused the corrupted
  blocks, causing heap metadata to appear in pipe output

**Why 2-stage pipes worked:** The same bugs existed but with fewer heap
cycles — the freed memory hadn't been reused yet, so writes landed
harmlessly. With 3+ stages, the extra child's allocations reused the
freed block, and corruption became visible.

### The Fix

**File:** `apps/dash/jobs.c`, function `forkshell()`

Set `vforked = 1` before calling `fork()` so that `forkchild()` takes
the early-return path (line 900: `if (lvforked) return;`), skipping
`forkreset()` and `freejob()`. This is exactly what `vforkexec()` (the
other call site for vfork in dash) already does correctly.

```c
// Before (broken):
pid = fork();
if (pid == 0)
    forkchild(jp, n, mode);

// After (fixed):
vforked++;
pid = fork();
if (pid == 0) {
    forkchild(jp, n, mode);
} else {
    vforked = 0;
    forkparent(jp, n, mode, pid);
}
```

### Design Decision Called Out

**The `#define fork() vfork()` pattern is inherently fragile.** Any code
path that modifies parent-owned data between `vfork()` and `exec()`
will corrupt the parent. The `vforked` flag is a manual guard that must
be set at every call site. This was done correctly in `vforkexec()` but
missed in `forkshell()` — the exact kind of bug this pattern invites.

A better long-term fix would be either:
1. Implement a minimal `fork()` using the existing allocator (copy
   user memory on fork)
2. Make the kernel's vfork properly isolate the child's heap from
   the parent's (e.g., snapshot the malloc state)

But both are expensive. The `vforked` flag fix is correct and sufficient
for now, since all fork() call sites in dash are now covered.

## Bug 2: `rmdir` command not found

### Symptoms

```
# rmdir tmp
/bin/dash: 4: rmdir: not found
```

### Root Cause

The `rmdir()` **syscall** was fully implemented (`SYS_RMDIR #40`,
`fs_rmdir()` in `kernel/fs.c`), and `rm -r` used it internally. But no
standalone `/bin/rmdir` binary existed. Users expected `rmdir` to work
as a command, since `mkdir` existed.

### The Fix

Created `apps/rmdir.c` — a minimal standalone command matching the style
of `apps/mkdir.c`. Added it to:
- `apps/Makefile` PROGRAMS list
- Top-level `Makefile` CORE_BINS list
- `.gitignore`

### Design Note

This was a simple omission, not a design flaw. The project added `mkdir`
as a command but never added its counterpart. Checked the other expected
POSIX utilities — `chmod`, `chown`, `ln` are also missing but those
don't have kernel support yet either.

## Bug 3: `cat` error message omits filename

### Symptoms

```
# cat nonexistent
cat: cannot open file
```

The error message didn't say *which* file couldn't be opened. When
running `cat` on multiple files, this made it impossible to know which
one failed.

### Root Cause

`cat.c` used a hardcoded error string with a fixed length:
```c
const char *msg = "cat: cannot open file\n";
write(2, msg, 22);
```

This avoided `#include <stdio.h>` but sacrificed usability.

### The Fix

Replaced the raw `write()` with `fprintf(stderr, ...)` that includes
`argv[i]` (the filename). Also changed `cat` to continue processing
remaining files on error (previously returned immediately on first
error), and added proper includes.

## Bug 4: `more` with no args causes "Out of space"

### Symptoms

```
# more
/bin/dash: 2726Out of space
```

### Root Cause

This was a **secondary effect of Bug 1** (vfork heap corruption). After
running pipe commands that triggered the heap corruption, dash's
internal allocator (`stalloc()`, `ckmalloc()`) would fail because the
heap metadata was corrupted. The "2726" in the output was a corrupted
line number variable.

### The Fix

No separate fix needed. Fixing Bug 1 (the vfork+forkshell corruption)
prevents the heap corruption that caused this failure. After the fix,
`more` with no args works correctly (reads from stdin in interactive
mode).
